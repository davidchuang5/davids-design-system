# enterprise-design-system

A React design system that is generated, tested, and documented by a multi-agent AI pipeline instead of being hand-written component by component.

This repo contains two things:

1. **The design system itself** — a published-ready npm package of accessible React components (`src/`).
2. **The agent pipeline that builds it** (`src/orchestrator.ts` and `src/agents/`) — an orchestrator that delegates to specialized subagents (tokens, components, tests, stories) using the [Claude Agent SDK](https://www.npmjs.com/package/@anthropic-ai/claude-agent-sdk).

The interesting part of this project isn't any single component — it's the pipeline that produces them consistently, with tests and documentation included, every time.

---

## Why this exists

Most design system tutorials stop at "here's a Button component." This project is an exploration of a different question: **can an agentic workflow reliably produce production-grade frontend code with the same rigor a senior engineer would apply by hand** — correct ARIA, typed props, CSS module integrity, test coverage, and Storybook documentation — without a human writing each file?

It's also a demonstration of agent architecture patterns that generalize well beyond design systems: orchestrator/subagent delegation, human-in-the-loop checkpoints, per-task context isolation, and cost-aware model selection.

## Architecture

```
                ┌─────────────────┐
                │   Orchestrator    │   claude-opus-4-7
                │ (per component)   │
                └─────────┬────────┘
                          │ Task tool delegation
        ┌─────────────────┼─────────────────┐
        ▼                 ▼                 ▼
┌───────────────┐ ┌───────────────┐ ┌───────────────┐
│ component-agent│ │  test-agent   │ │storybook-agent│
│  (sonnet)      │ │  (haiku)      │ │  (haiku)      │
└───────────────┘ └───────────────┘ └───────────────┘
```

A separate, earlier phase runs **token-agent** (sonnet) in isolation to generate the design tokens (`tokens.json`, `variables.css`), with a human checkpoint before the pipeline proceeds to components.

Each component is then built in its own isolated agent run: the orchestrator invokes `component-agent` first, and once that succeeds, fans out to `test-agent` and `storybook-agent` in parallel. Components are processed one at a time, each with a fresh context — this keeps token usage predictable and prevents one component's generation history from bleeding into the next.

### Why subagents instead of one big prompt

Splitting responsibilities into narrow, single-purpose agents (one for tokens, one for component code, one for tests, one for stories) makes each agent's prompt focused and its output easy to validate. It also means failures are isolated: if `storybook-agent` fails for one component, the component and its tests are still usable, and the orchestrator reports the failure rather than discarding everything.

## What gets generated

For every component in the spec, the pipeline produces:

| File                        | Generated by    |
| --------------------------- | --------------- |
| `ComponentName.tsx`         | component-agent |
| `ComponentName.module.css`  | component-agent |
| `ComponentName.d.ts`        | component-agent |
| `ComponentName.test.tsx`    | test-agent      |
| `ComponentName.stories.tsx` | storybook-agent |

Plus, once per run:

| File                         | Generated by                                         |
| ---------------------------- | ---------------------------------------------------- |
| `tokens/variables.css`       | token-agent                                          |
| `tokens/tokens.json`         | token-agent                                          |
| `tokens/tokens.js` + `.d.ts` | local script (`generate-tokens.ts`), not an LLM call |

## Running the pipeline

```bash
npm install
cp .env.example .env   # add your ANTHROPIC_API_KEY
npx tsx src/index.ts
```

Edit the spec in `src/index.ts` to change which components get built, the color palette, and the border radius style:

```ts
const spec: DesignSystemSpec = {
  name: 'nova-ui',
  components: ['Button', 'Input', 'Badge', 'Modal', 'Tooltip'],
  colorPalette: 'neutral grays with an indigo primary accent',
  borderRadius: 'rounded',
};
```

The pipeline will:

1. Skip any component that already has a folder under `src/components/` (delete the folder to force regeneration).
2. Generate tokens, then **pause for confirmation** so you can review `variables.css` before components are built against it.
3. Build, test, and document each remaining component in sequence.
4. Update `src/index.ts`'s barrel export with the new components.
5. Write a `run-log.json` summarizing every phase, including per-component token usage and cache hit rate.

## Cost and model strategy

Different agents use different models based on how much reasoning the task actually requires:

- **Orchestrator** — `claude-opus-4-7`, since it coordinates multi-step delegation and needs to reliably sequence dependent work.
- **token-agent / component-agent** — `claude-sonnet-4-6`, since these require correct, non-trivial output (real ARIA semantics, real TypeScript types).
- **test-agent / storybook-agent** — `claude-haiku-4-5`, since these are comparatively templated tasks.

Each agent is also scoped to the minimum tool access it needs (`Write`/`Read` only for token-agent, no `Bash` unless a step genuinely requires running a build command), and `run-log.json` tracks cache hit rate so regressions in prompt caching are visible immediately.

## Project structure

```
enterprise-design-system/
├── src/
│   ├── orchestrator.ts        # main pipeline entry point
│   ├── generate-tokens.ts     # tokens.json → tokens.js/.d.ts (no LLM call)
│   ├── agents/
│   │   ├── definitions.ts     # AgentDefinition objects (tools, model, prompt ref)
│   │   └── prompts.ts         # system prompt strings per agent
│   ├── types.ts
│   ├── tokens/                # generated — see Running the pipeline
│   ├── components/             # generated — one folder per component
│   └── index.ts                # barrel export, auto-updated by the pipeline
├── run-log.json               # generated after each run
└── .env                       # ANTHROPIC_API_KEY (not committed)
```

## Status

This is an active portfolio/learning project, not a published package yet. Components are generated against an evolving prompt set — expect to review and occasionally hand-correct output, particularly for components with complex interaction patterns (focus trapping, portal rendering) that are harder for an agent to get exactly right on the first pass.
